# Задача 1-3

## Описание решения домашнего задания

Немного затянул с выполнением, но в дедлайн уложился. В ходе работы мне хотелось не просто выполнить задачу, но и применить на практике некоторые теоретические концепции, чтобы лучше их закрепить.

### Ключевые моменты реализации:
1. Структура проекта (Enum):
Вместо использования коллекций (ArrayList) для хранения путей к файлам и директориям, я решил использовать enum GameStructure. Списки показались мне слишком «захардкоженным» и менее типобезопасным решением. Enum позволил четко структурировать пути и свойства (файл это или директория) в одном месте.
2. Логирование (BufferedWriter вместо StringBuilder):
Я сознательно отказался от накопления логов в StringBuilder, как было в примере. Мне показалось, что при больших объемах данных это может привести к OutOfMemoryError. Вместо этого я реализовал потоковое логирование через FileWriter, обернутый в BufferedWriter.
3. Архитектура (Класс Game):
Для первой задачи был создан класс Game, который инкапсулирует логику создания файловой структуры и работы с логгером. Это позволило отделить управление файловой системой от данных.
4. Восстановление данных (Task 3):
В третьей части реализована десериализация последнего сохранения. С помощью Stream API я нахожу самый свежий *.dat файл (после распаковки архива) и восстанавливаю из него объект GameProgress.
